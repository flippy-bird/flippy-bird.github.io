发表于2026-03-30|更新于2026-06-15
|总字数:1.6k|阅读时长:4分钟

向量数据库索引

Flat Index (暴力搜索)

Flat 索引实际上不进行任何特定的处理，直接存储原始向量，接收查询时，会计算查询向量与数据库中每一个向量的距离（如欧氏距离或余弦相似度），然后排序取 Top-K。

优点： 无任何精度损失。

缺点： 在大规模数据上，查询速度慢。

适用场景： 数据量极小（如几万条以内）或对精度要求极高的场景。

IVF (Inverted File Index / 倒排文件索引)

原理： IVF 基于聚类思想。

  • 训练阶段：使用 K-Means 算法对向量进行聚类，划分成n个簇（Cluster），每个簇有一个中心点。
  • 存储阶段：将每个向量划分到距离其最近的簇中。
  • 查询阶段：先计算查询向量与所有簇中心点的距离，找到距离最近的k个簇，然后仅在这些簇包含的向量中进行搜索。

优点： 缩小搜索范围，查询速度快。

缺点： 属于近似搜索，边界点的向量可能会被漏掉，会有精度损失。

适用场景： 中等规模数据，追求速度与精度的平衡。

PQ (Product Quantization)

原理： PQ 是一种数据压缩技术，常与 IVF 结合使用（IVF_PQ）。

  • 训练阶段
    • 切分：将高维向量切分为n个子向量（如将128 维向量切分成4个32 维的子向量）。
    • 聚类：对每一组子向量分别进行 K-Means 聚类，生成Codebook。
  • 编码阶段
    • 切分：将原始的连续浮点数向量，切分为n个子向量。
    • 编码：获取离每个子向量最近的聚类中心，将其替换为对应的聚类中心ID，高维浮点数向量被压缩成低维int型向量进行存储。每个向量最终存储的是n个聚类中心ID。
  • 查询阶段
    • 切分：将查询向量切分为n个子向量。
    • 查询：每个子向量预先和256个聚类中心计算距离（得到距离表），库中每个向量根据其所有的向量值（聚类中心的索引）查表获取距离，然后相加即为最终的距离。

优点： 内存占用极低，计算速度极快（查表）。

缺点： 精度损失较大（因为是近似表示），可大量召回，然后使用重排序提升召回精度。

适用场景： 内存受限、数据量巨大的场景。

HNSW (Hierarchical Navigable Small World)

原理： HNSW结合了跳表（Skip List）和小世界网络（NSW）的思想。

跳表：

假设在一个有序的数字链表中查找数字 85： 1 -> 2 -> ... -> 50 -> ... -> 80 -> 85 -> ... -> 100

  • 普通链表：从 1 开始，一个一个往后找，非常慢。
  • 跳表：建立分层索引。
    • 顶层：只存 1 -> 50 -> 100（稀疏，跳得远）。
    • 中层：存 1 -> 25 -> 50 -> 75 -> 100。
    • 底层：存所有数字。
    • 查找过程：
      • 先在顶层找到 50，发现 100 大了
      • 降级到中层，从 50 走到 75，发现 100 又大了
      • 再降级到底层，从75开始查找。走两步找到 85。

NSW:

将所有的向量连接成一张图，每个向量作为一个节点。

节点的邻居应该既有相似度高的（短程连接），又有相似度低的（长程连接）。如果只连接相似度高的，在搜索时限制了探索效率，容易陷入到局部最优（局限在固定的范围内）。相似度低的边，可以增加搜索时的跳转范围，探索更大的可能性。

  • 如何构建图：每次插入节点时找已插入节点中的最相似的M个节点连接边。
  • 好处：
    • 整个图是联通的
    • 天然存在短程连接和长程连接（当前插入节点的时候因为图是不完整的，所以当前连接的最近邻并不一定是完整图中该节点的最近邻，在完整图中可能还离该节点较远，所以说天然存在短程连接和长程连接）
  • 缺点：插入顺序会影响长程连接和短程连接，先插入的节点长程连接较多，后插入的节点长程连接较少

结构：构建分层图结构。最底层（Layer 0）包含所有节点，是高密度的近邻图。越往上层，节点越少，连接越稀疏。上一层是下一层的子图。

查询：

先在顶层进行初筛，将该层搜索到的节点作为下一层的入口节点，逐层搜索，最后一层的topk作为最终搜索结果。

假设要找一个目标向量 Query：

  • 顶层：进入 Layer 2**，**这里的节点很少，通过贪婪搜索，迅速找到一个离 Query 最近的向量C。
  • 中间层：进入 Layer 1，向量C作为入口节点，在C的近邻中继续搜索，找到D
  • 底层：进入 Layer 0，向量D作为入口节点，在D的近邻中继续最后的精细搜索，找到最终的 Top-K 结果。

优点： 查询性能和召回率都很高。

缺点： 内存占用较大（需要存储图的边关系），构建索引慢，不支持高效的动态增删。

适用场景： 对性能和召回率要求都很高，且内存充足的场景。

总结

索引类型 查询速度 召回率 (精度) 内存消耗 构建速度
Flat 慢 100% 低 (仅存原始向量) 无 (无需构建)
IVF 快 高 (取决于 nprobe) 中 快
PQ 极快 中/低 (有损) 极低 中
HNSW 极快 极高 高 (存邻接表) 慢

建议：

  • 如果内存够用，优先选 HNSW，速度和精度都比较高。
  • 数据量特别大且内存紧张，选 PQ，先大量召回再重排序。
  • 数据量小（几千/几万），选Flat。
